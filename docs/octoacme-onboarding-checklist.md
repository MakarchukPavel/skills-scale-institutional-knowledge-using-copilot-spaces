## Purpose
Provide a structured onboarding experience for new team members joining OctoAcme projects, ensuring they have the context, access, and support needed to contribute effectively.

## When to Use
- When a new team member joins the project
- When an existing team member transitions to a new role
- When contractors or external collaborators join for specific engagements

---

## General Onboarding Checklist (All Roles)

### Day 1: Welcome and Orientation
- [ ] Welcome meeting with Project Manager or team lead
- [ ] Review team composition and org chart
- [ ] Introduction to key stakeholders and team members
- [ ] Overview of project goals, timeline, and current status
- [ ] Review of OctoAcme project management documentation:
  - [ ] [Project Management Overview](octoacme-project-management-overview.md)
  - [ ] [Roles and Personas](octoacme-roles-and-personas.md)
  - [ ] [Communication and Escalation Guide](octoacme-communication-escalation-guide.md)

### Week 1: Access and Environment
- [ ] Grant access to required systems and tools:
  - [ ] GitHub repository access (appropriate permissions)
  - [ ] Project board access (GitHub Projects or equivalent)
  - [ ] Communication channels (Slack, Teams, etc.)
  - [ ] Documentation (Confluence, Notion, or repo docs)
  - [ ] Development environment setup (if applicable)
- [ ] Add to relevant meeting invites:
  - [ ] Daily standups
  - [ ] Sprint planning and backlog refinement
  - [ ] Weekly team syncs
  - [ ] Retrospectives
- [ ] Review current sprint backlog and priorities
- [ ] Shadow a team ceremony (standup, planning, or review)

### Week 1-2: Process and Workflow
- [ ] Review team conventions and working agreements
- [ ] Understand branching strategy and PR workflow
- [ ] Review Definition of Done and acceptance criteria standards
- [ ] Understand escalation paths and how to raise blockers
- [ ] Complete any required compliance or security training

### Week 2-4: Integration and First Contributions
- [ ] Pair with a buddy or mentor for guidance
- [ ] Complete first small task or contribution
- [ ] Attend first retrospective and provide feedback
- [ ] 30-day check-in with manager or team lead

---

## Role-Specific Onboarding Tasks

### Developers
- [ ] Local development environment setup complete
- [ ] Review codebase architecture and key components
- [ ] Understand testing strategy and CI/CD pipeline
- [ ] Review code style guide and linting rules
- [ ] Complete first code review (as reviewer)
- [ ] Complete first PR (as author)
- [ ] Shadow a production deployment (if applicable)

### QA / Testing
- [ ] Access to test environments and test data
- [ ] Review test plan and test case repository
- [ ] Understand bug reporting process and tools
- [ ] Review acceptance criteria standards
- [ ] Complete first test cycle or bug verification
- [ ] Understand release testing and sign-off process

### Scrum Master
- [ ] Review team's Agile practices and ceremonies
- [ ] Understand team velocity and historical performance
- [ ] Review current blockers and impediments
- [ ] Meet 1:1 with each team member
- [ ] Facilitate first ceremony (standup or refinement)
- [ ] Review and understand escalation procedures

### Business Analyst
- [ ] Review current requirements documentation
- [ ] Understand stakeholder map and communication preferences
- [ ] Review backlog and upcoming features
- [ ] Shadow a stakeholder meeting or discovery session
- [ ] Review requirements traceability approach
- [ ] Complete first user story or requirements document

### UX Designer
- [ ] Review design system and style guide
- [ ] Access to design tools (Figma, Sketch, etc.)
- [ ] Review current designs and user research
- [ ] Understand design review and approval process
- [ ] Meet with Product Manager on design priorities
- [ ] Complete first design review or critique session

### Tech Lead
- [ ] Deep dive on architecture and technical decisions
- [ ] Review technical debt backlog
- [ ] Understand security and compliance requirements
- [ ] Review Architecture Decision Records (ADRs)
- [ ] Meet 1:1 with each developer
- [ ] Conduct first code review
- [ ] Review incident response procedures

### Product Manager
- [ ] Review product roadmap and vision
- [ ] Understand customer personas and market context
- [ ] Review current backlog and prioritization criteria
- [ ] Meet with key stakeholders
- [ ] Review success metrics and dashboards
- [ ] Shadow a customer call or user research session

### Project Manager
- [ ] Review project charter and current status
- [ ] Understand risk register and mitigation strategies
- [ ] Review stakeholder communication plan
- [ ] Meet with Product Manager for alignment
- [ ] Review budget and resource allocation (if applicable)
- [ ] Take ownership of weekly status reporting

## Onboarding Success Criteria

By end of Week 4, the new team member should be able to:
- [ ] Describe the project's goals, stakeholders, and current status
- [ ] Navigate all required tools and systems independently
- [ ] Understand and follow team workflows and conventions
- [ ] Contribute meaningfully to their role's responsibilities
- [ ] Know who to ask for help and how to escalate blockers
- [ ] Participate actively in team ceremonies

---

## Onboarding Feedback

After 30 days, gather feedback from the new team member:
- What went well in the onboarding process?
- What information was missing or hard to find?
- What would have helped you get up to speed faster?
- How can we improve the onboarding experience?

Use this feedback to continuously improve the onboarding process in retrospectives.
